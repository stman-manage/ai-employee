Flag A — visible-browser scope. BrowserAct stealth browsers run on BrowserAct's cloud, not inside the Render container — so noVNC will not show them. noVNC will only show a local Chromium (--type chrome) launched inside the container under Xvfb. For a live view of a hosted stealth session, BrowserAct's own remote-assist --objective "…" generates a live URL we can embed instead. Decision needed: local-Chromium-in-container (full noVNC control, less stealth) vs. hosted stealth + remote-assist embed (best stealth, view only on demand). I recommend running local Chromium in the container for the visible pane and reserving stealth for anti-bot sites.

03Render Free — Docker runner + visible browser

RequirementStatusNotes
Free Docker web serviceYESFree instances support Docker (Dockerfile build or prebuilt image). No credit card, does not expire.
Outbound HTTPSYESAllowed (to BrowserAct cloud + Cloudflare). Counts toward monthly free bandwidth.
noVNC streamingYESnoVNC is just HTTP + WebSocket on the service's public port. Render free services accept inbound WebSockets. Xvfb + x11vnc + websockify + noVNC all run inside the container.
Subprocesses (CLI, Chromium)YESFull container — supervisord can run Xvfb, Chromium, x11vnc, websockify and the Node API together.
Flag C — free-tier physics. Three real limits to design around: (1) One public port only — the Node server must bind 0.0.0.0:$PORT (default 10000) and serve /health, /run-tool, and proxy /vnc WebSockets all on that single port. (2) Spin-down after 15 min idle → cold start of up to ~1 min on the next request; an open noVNC WebSocket keeps it warm. (3) 512 MB RAM — Chromium + Xvfb + noVNC is tight; we run a single low-res Chromium (e.g. 1280×720) and one tab to stay within it. HTTP Basic Auth gates /vnc; the RUNNER_INTERNAL_TOKEN header gates /run-tool.

04Proposed Dockerfile (draft — not for execution yet)

# Render free Docker service: BrowserAct CLI + visible Chromium + noVNC
FROM node:20-bookworm-slim

ENV DEBIAN_FRONTEND=noninteractive DISPLAY=:99 SCREEN=1280x720x24

# System: Chromium, virtual display, VNC bridge, noVNC, python+uv for the CLI
RUN apt-get update && apt-get install -y --no-install-recommends \
      chromium xvfb x11vnc novnc websockify \
      python3 python3-pip curl ca-certificates supervisor tini \
    && curl -LsSf https://astral.sh/uv/install.sh | sh \
    && /root/.local/bin/uv tool install browser-act-cli --python 3.12 \
    && rm -rf /var/lib/apt/lists/*

ENV PATH="/root/.local/bin:/root/.local/share/uv/tools/browser-act-cli/bin:$PATH"

WORKDIR /app
COPY package*.json ./
RUN npm ci --omit=dev
COPY . .

# Single public port (Render sets $PORT, default 10000); 0.0.0.0 bind.
# Node server: /health, /run-tool (token auth), and proxies /vnc (basic auth) -> websockify.
EXPOSE 10000
ENTRYPOINT ["/usr/bin/tini","--"]
CMD ["/usr/bin/supervisord","-c","/app/supervisord.conf"]

supervisord runs: Xvfb :99 → chromium --no-sandbox (or launched on demand by the CLI) → x11vnc → websockify/noVNC → node server.js. On boot, server runs browser-act auth set $BROWSERACT_API_KEY. Exact files come in Step 4.

05High-level system blueprint

Job lifecycle: webhook → Worker creates a D1 job → agent loop calls GLM-4.7-Flash → model emits a tool call → Worker POSTs /run-tool on Render (token-authed) → CLI acts in Chromium → screenshot/state returned → step logged to D1 → streamed to the dashboard over SSE → repeat until final_answer or ask_user. The live Chromium is watchable in real time via the embedded noVNC pane. Nothing runs on your machine; nothing outside BrowserAct is paid.

06Open decisions before Step 2

  • Visible browser mode (Flag A): local Chromium in container for full noVNC control (my recommendation), vs. hosted stealth + remote-assist embed for max anti-bot resilience.
  • Event transport (Flag B): SSE on a plain Worker (simpler, fully free — my recommendation) vs. WebSockets via a Durable Object.
  • Storage: D1 (recommended) vs. KV. I'll proceed with D1 unless you prefer KV.
  • Runner idle/cold-start (Flag C): accept ~1 min cold start on the free tier, or add a lightweight keep-alive ping (still free). I'll accept cold starts by default.
